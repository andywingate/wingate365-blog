<table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"></td></tr><tr><td class="tr-caption" style="text-align: center;">..there, I said it!</td></tr></tbody></table><p>Approvals are a vital business control process, the most common area being around the approval for purchases and expenditure.</p><p>Dynamics 365 Business Central (D365BC) has a built-in approval workflow engine and can also tap into Power Automate Approvals.&nbsp;</p><p>Approvals via Power Automate have a few really great positives, but this post is not about those. Here I'd like share some tips on using BC's built-in approval engine for some complex scenarios.&nbsp;</p><p>For really complex approvals, Power Automate gets tricky quickly. While the build-in approval can scale stably in my experience.</p><p>To illustrate my view, here is a complex approval scenario that the BC engine handles fairly easily.&nbsp;</p><h2 style="text-align: left;">Multi-Approval Example &amp; How-to</h2><h3 style="text-align: left;">Business Requirements</h3><div style="text-align: left;">The Purchaser Admin (PA) will create a draft PO, which in all cases must be approved by the Project Manager (PM). Following the PM approval further approval is value based as shown below.</div><div style="text-align: left;"><ul style="text-align: left;"><li>Level 1 - approval by PM up to £10k</li><li>Level 2 - further approval by a named Senior Manager up to £50k</li><li>Level 3 - further approval by a named Head of Team up to £100k</li><li>Level 4 - further approval by a named Director up to £250k&nbsp;&nbsp;</li><li>Level 5 - further approval by the CEO for over £250k</li></ul><div>There are multiple departments, which is held on the document as a global dimension.&nbsp;</div><div><br /></div><div>Quotes and Orders over 250k, in addition to the above logic, also need to go to a named user in the procurement team, but this is not necessary for invoices.</div><h3 style="text-align: left;">Workflow Solution</h3><div><ul style="text-align: left;"><li>Level 1 - the Project Manager are all created as Purchasers, so the PA who could be working across multiple projects can select the appropriate project manager, i.e., they drive the initial step by selecting who to send the document to.</li><li>Level 2..5 - These will be managed with the standard Workflow User Groups, the department global dimension is used to help route the approvals.</li><li>We will create 1 workflow per department using the "Send Another Approval" approach that I explain below.</li></ul><h3 style="text-align: left;">Technical Setup</h3></div><div>Here is an example workflow:</div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; To manage this you need to tweak the <b>Workflow Event/Response combinations</b>. By default not all the Then responses shown below are available after the <b>When Event</b> "An approval request is approved"</div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; Because when you are testing, it is super handy to know where things are routing. This is a huge help when debugging the config.</li><li><b><i>Vital:</i></b> The flux capacitor for multi-approval workflows is the use of the &lt;(Optional) Select Next Step&gt;. This is available from the last step of the <b>Then responses. </b>To make multi-approvals work you need to set the next step to the last Then response of line 1.</li></ul></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; If you don't remember to do this small config, things like the Cancel Approval, Delegate, Reject etc etc won't work.&nbsp; &nbsp; &nbsp;</div><h4 style="text-align: left;">Lines 9 &amp; 11 - Send to next approver or Release the Doc</h4><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjfpD8AZAEGQqv9uacymAxRnfcskWLYZYD4i100a92ibWkRJWjJilqYCzNegI4ZDv8UkdjgubXiXEJbsONb54dZkDdFhaoTVZFq_QQc6HnwOCmpX4E67F1ti-nU4RKPerNXMt2kiJvN2a7GS9eZy7ufOOZUOI16uWHw3hpH54U0R5JnnoFkpatc/s981/2024-06-01_14h58_15.png" style="margin-left: auto; margin-right: auto;"><img border="0" data-original-height="396" data-original-width="981" height="258" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjfpD8AZAEGQqv9uacymAxRnfcskWLYZYD4i100a92ibWkRJWjJilqYCzNegI4ZDv8UkdjgubXiXEJbsONb54dZkDdFhaoTVZFq_QQc6HnwOCmpX4E67F1ti-nU4RKPerNXMt2kiJvN2a7GS9eZy7ufOOZUOI16uWHw3hpH54U0R5JnnoFkpatc/w640-h258/2024-06-01_14h58_15.png" width="640" /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">Lines 9 and 11</td></tr></tbody></table><div><br /></div><div>These lines hold the fairly normal "send to next person" / "we are done now" logic. The <b>On Condition </b>of "Sequence &gt;1" ensures they are not part of the initial approval by the PM.&nbsp;</div><div><br /></div><div>The line 9 "Send to next person"&nbsp;<b>Then responses</b> are&nbsp;&nbsp;&nbsp;</div><div><ul style="text-align: left;"><li>Send approval request for the record and create a notification.</li><li>Show message "Sent for further approval".</li><ul><li>&gt;Next then "&lt;the last <b>Then Response</b> of line 1&gt;"</li></ul></ul></div><div>The line 11 "we are done now" <b>Then responses</b> are</div><div><ul style="text-align: left;"><li>Remove record restriction.</li><li>Release the document.</li><li>Show message "Fully approved".</li><li>Create a notification for &lt;Sender&gt;.</li></ul></div><div>The other lines handle the different values and document types, but the principle remains the same for doing an direct approver approval followed by a workflow user group approval all inside 1 workflow!</div><h3 style="text-align: left;">Other departments?</h3><div>This workflow once fully tested was replicated for all the departments, using import and export you can do a fairly easy find and replace on the workflow user group codes. The only manual step is the selection of the department code for line 1 on Condition. So overall it's fairly trivial to copy the config for a lot of departments.</div><div><br /></div><div>I could have added the other departments inside this workflow, but it was faster to copy the whole workflow and gave some granularity around testing (enabling/disabling for a give single department)&nbsp;</div><h3 style="text-align: left;">General workflow tips</h3><div>Setting up and testing workflows can be a pain - always create a workflow testbed system - where you have multiple logins for all the personas in your workflow, use the edge profiles so you can easily have 5 edge browsers open / pinned to your task bar for each of the different user and quickly test a super complicated approval going to multiple accounts without actually needing to tie up anyone else.</div><div><br /></div><div>When doing multiple approvals it is possible to create an infinite loop. Be sure to use the "Sequence No" in the On Condition to avoid this. If you do have an infinite loop event - end your session from the admin portal!</div><div><br /></div><div>Always insist on a clear flow diagram for the approval logic - that must always be creased by the end client. This is a vitally important document for creating and testing approval flows. Also allows you to judge the time to do the work and acts as a CR trigger if anyone changes their mind on the eve of go-live for example!&nbsp;&nbsp;</div><h3 style="text-align: left;">Next time</h3><div>Here is another complex approval requirement!
